# Agent评估体系建设初稿

## 1. 建设目标

这套评估体系当前要解决 5 件事：

1. 给产品、研发、测试、安全统一一套评估范围和判断口径。
2. 把 Agent 项目从需求到上线的关键评审节点定下来。
3. 明确每个阶段要检查什么、产出什么、谁负责。
4. 让质量验收和安全验收不再只看“功能是否跑通”。
5. 为后续沉淀模板、样本、评分器和周运行机制提供统一入口。

## 2. 适用范围

适用于以下类型的 Agent 产品：

- 单 Agent 问答、诊断、规划、执行类产品
- 带工具调用的任务型 Agent
- 多节点协同的 Multi-Agent 工作流
- 面向内部提效、运营辅助、知识问答、测试生成、问题诊断的 Agent 能力

当前不包含以下范围：

- 基座模型选型和采购决策
- 模型训练和微调方案
- 独立的安全规范体系正文
- 发布平台或评测平台的系统建设方案

## 3. 体系定位

这套体系不是单一测试方案，也不是只在提测前使用的验收清单。它覆盖 4 类工作：

- 产品设计阶段的可评审要求
- 技术方案阶段的可实现、可观测、可控要求
- 测试和质量阶段的样本、评分器、回归要求
- 安全和交付阶段的风险拦截、人工接管、上线留档要求

现阶段先把“评什么、谁来评、产出什么”定清楚。自动化平台、发布门禁和全量量化指标后续再单独展开。

## 4. 为什么需要单独建立 Agent 评估体系

传统 Web 项目通常按页面、接口、数据库和确定性逻辑组织评审。Agent 项目除了功能实现，还涉及：

- 任务目标是否定义清楚
- 工具调用是否受控
- 多轮状态和共享 state 是否可靠
- 模型在不确定场景下是否会误判
- 结果是否有依据
- token 成本和人工接管率是否可接受
- 高风险动作是否有拦截、降级和回退

如果没有单独的评估体系，常见问题是：

- 产品设计阶段没有写清失败边界和人工接管点
- 技术方案阶段没有落实日志、轨迹、state 快照和工具权限
- 研发阶段只看功能完成，不看可复现和可定位
- 验收阶段只看跑通率，不看坏例、风险和成本

## 5. 阶段划分

| 阶段 | 目标 | 主要问题 |
| --- | --- | --- |
| G1 产品设计评审 | 把任务目标、范围边界、失败边界定清楚 | 这个 Agent 要解决什么问题，不能做什么 |
| G2 技术方案评审 | 把实现路径、状态设计、工具边界、可观测性定清楚 | 怎么实现，怎么定位问题，怎么控制风险 |
| G3 研发实现过程检查 | 把评审要求落到开发过程和联调过程 | 是否按方案实现，是否具备评测前置条件 |
| G4 测试评估 | 用样本、评分器、回归结果判断效果和稳定性 | 是否达到质量要求，问题分布在哪里 |
| G5 质量交付验收 | 形成版本质量结论 | 是否具备进入下一阶段或上线的质量条件 |
| G6 安全交付验收 | 形成版本安全结论 | 是否存在越权、误写、注入、数据泄露等高风险问题 |
| G7 上线后运行复盘 | 把线上问题、坏例和成本变化回流体系 | 是否持续稳定，是否需要调整范围或策略 |

## 6. 各阶段评估重点

### 6.1 G1 产品设计评审

重点检查：

- 任务目标是否清楚
- 成功终态和失败终态是否清楚
- 自动化范围和人工接管范围是否清楚
- 高风险场景是否列出
- 单次任务成本上限是否给出

阶段产出：

- 成功标准
- 失败边界
- 风险分级
- 人工接管规则
- 成本预算口径

### 6.2 G2 技术方案评审

重点检查：

- 输入、输出、状态字段是否定义清楚
- 工具调用范围、权限范围、禁止动作是否落到方案
- 是否记录输入、提示词、工具参数、工具返回、状态快照
- 高风险动作是否有只读模拟、二次确认、回退方案
- 外部接口空数据、脏数据、超时是否有处理策略
- token 成本和工具调用成本是否可统计

阶段产出：

- 技术方案
- 工具权限和调用边界说明
- 日志与可观测性字段清单
- 状态设计说明
- 异常处理和回退说明

### 6.3 G3 研发实现过程检查

重点检查：

- 方案要求是否已经落到实现
- 关键字段、日志、状态快照是否可导出
- 高风险动作保护是否已接入
- 灰度开关、人工接管、失败重试是否可用
- 是否具备进入测试评估的前提条件

阶段产出：

- 联调记录
- 可测性检查结果
- 版本配置和依赖说明

### 6.4 G4 测试评估

重点检查：

- 是否覆盖正常、边缘、混沌、记忆四类样本
- 是否覆盖线上翻车、线上发现、故障注入样本
- 是否区分代码评分器、状态评分器、工具评分器、LLM 评分器
- 是否记录主流程通过率、P0 失败数、token 成本、人工接管率
- 问题是否按推理层、工具层、API 层、状态层归类

阶段产出：

- Eval 样本集
- 评分器清单
- 轮次结果
- 坏例清单
- 复盘记录

### 6.5 G5 质量交付验收

重点检查：

- 主流程是否达到目标通过率
- 是否存在未关闭的 P0 风险
- 失败问题是否有明确归因和修复计划
- 版本是否具备复测和回归能力
- 质量结论是否满足进入下一阶段的条件

阶段产出：

- 质量验收结论
- 未关闭问题清单
- 版本风险说明

### 6.6 G6 安全交付验收

重点检查：

- 是否存在越权调用工具或访问超范围数据
- 是否存在提示注入导致的越界行为
- 是否存在错误写入、误操作、误触发外部动作
- 是否存在敏感信息泄露风险
- 高风险场景是否有人工确认、回退或禁用策略

阶段产出：

- 安全验收结论
- 高风险问题清单
- 上线前限制条件

### 6.7 G7 上线后运行复盘

重点检查：

- 线上翻车是否已入库
- 用户追问率、人工接管率、失败重试率是否异常
- token 成本是否明显上升
- 哪类问题重复出现
- 是否需要收缩范围、调整策略或补充样本

阶段产出：

- 周运行记录
- 坏例回流记录
- 阶段复盘记录

## 7. 阶段、产出、责任人总表

| 阶段 | 主要产出 | 主责 | 参与方 | 进入下一阶段前需要确认 |
| --- | --- | --- | --- | --- |
| G1 产品设计评审 | 成功标准、失败边界、风险分级、人工接管规则 | 产品 | 测试、开发、业务 | 任务目标、边界、风险、成本已确认 |
| G2 技术方案评审 | 技术方案、状态设计、工具边界、日志字段清单 | 开发 | 产品、测试、算法、安全 | 可观测性、回退、成本统计路径已确认 |
| G3 研发实现过程检查 | 联调记录、可测性检查结果、版本说明 | 开发 | 测试、算法 | 已具备样本执行和问题定位条件 |
| G4 测试评估 | 样本集、评分器清单、轮次结果、坏例清单 | 测试 | 产品、开发、算法 | 质量结果和坏例归因已形成 |
| G5 质量交付验收 | 质量验收结论、风险说明 | 测试/质量 | 产品、开发 | P0 风险是否关闭，是否允许推进 |
| G6 安全交付验收 | 安全验收结论、限制条件 | 安全 | 产品、开发、测试 | 是否允许进入上线候选 |
| G7 上线后运行复盘 | 周运行记录、坏例回流、复盘记录 | 测试/质量 | 产品、开发、业务 | 是否需要调整范围、样本和策略 |

## 8. 角色分工

| 角色 | 主要责任 |
| --- | --- |
| 产品 | 定义任务目标、边界、失败标准、人工接管范围、业务风险 |
| 开发 | 实现工具边界、状态设计、日志字段、回退能力、成本统计 |
| 测试/质量 | 组织样本、评分器、回归、坏例回流、质量验收 |
| 算法 | 支持提示词、模型策略、工具路由、LLM 评分器调优 |
| 安全 | 识别越权、注入、数据泄露、误操作等风险，给出验收意见 |
| 业务方 | 确认任务价值、坏例影响、人工接管规则 |
| 管理层 | 确认范围、投入节奏和阶段推进决策 |
