# Agent测试全流程方法论

## 1. 使用目的

本文件用于支持团队从需求评审开始组织 Agent 测试。重点覆盖 5 个问题：

- 产品需求是否适合做 Agent，而不是普通工作流或规则系统
- 测试应从哪个阶段介入，前置准备哪些可测性条件
- Eval 样本、评分器、分层测试和上线准入如何组织
- Token、人工、工具调用成本如何纳入 ROI 评估
- 上线后的失败样本、成本漂移和能力回退如何进入持续优化

## 2. 适用范围

适用于以下场景：

- 单 Agent 问答、诊断、规划、执行类系统
- 带工具调用的任务型 Agent
- 有共享状态和节点协同的 Multi-Agent 工作流
- 面向内部提效、知识问答、运营辅助、测试生成、问题诊断的企业场景

当前范围不包含以下内容：

- 基座模型选型方法
- 模型训练或微调方案
- 发布门禁平台实现细节

## 3. 执行要求

### 3.1 从需求和业务回报开始

先判断任务目标、错误代价、人工替代空间和成本上限，再决定是否投入 Agent 方案。没有明确业务收益和失败边界的场景，先不做大规模评测。

### 3.2 同时检查结果、过程、状态、成本

Agent 测试不能只看回答是否像对的，还要检查：

- 是否完成任务目标
- 是否使用了正确工具和参数
- 中间状态是否可追踪
- 失败时是否中止、降级或求助
- 单任务成本是否在预算内

### 3.3 评估真实风险，不追求用例数量

优先收集真实用户请求、线上翻车案例、监控异常和故障注入场景。低价值的标准问法样本可以少，P0 链路和高损失场景必须覆盖。

### 3.4 代码评分器优先，LLM 评分器补充

能用规则、状态检查、工具日志判断的维度，优先用代码评分器。LLM 评分器用于语义完整性、结论合理性、表达是否满足约束等代码难以覆盖的部分。

### 3.5 测试设计前置到架构阶段

日志、状态快照、工具调用记录、节点接口契约、版本标识、样本字段，如果等到联调后才补，测试成本会明显升高，很多问题也无法稳定复现。

## 4. 流程总览

| 阶段 | 目标 | 关键动作 | 主要产出 |
| --- | --- | --- | --- |
| S0 需求评审 | 判断是否值得做 Agent | 识别任务目标、风险、ROI、成本上限 | 成功标准、边界、预算口径 |
| S1 方案设计 | 建立可测性 | 定义状态、日志、工具契约、节点契约 | 可测性清单、运行字段 |
| S2 样本建设 | 建立可复跑样本池 | 收集真实任务、失败样本、故障注入样本 | Eval 数据集 |
| S3 评分器设计 | 建立判定机制 | 拆分代码评分器、状态评分器、LLM 评分器 | 评分器清单与提示词 |
| S4 分层执行 | 逐层验证能力 | 跑依赖、节点、任务、E2E、混沌回归 | 各层结果和缺陷 |
| S5 准入评审 | 判断是否进入下一阶段 | 综合效果、风险、成本、稳定性 | 准入结论和整改项 |
| S6 上线运行 | 采集真实反馈和成本漂移 | 监控失败、追问、人工接管、成本波动 | 周报、坏例入库、复盘 |

## 5. S0 需求评审

### 5.1 先回答 6 个问题

- 用户要解决的任务是什么，终态是什么
- 当前人工流程是什么，平均耗时和错误率是多少
- 这个任务是否依赖实时数据、外部工具或多轮状态
- 错误代价是什么，是否涉及资金、权限、对外承诺、生产变更
- 任务是否允许“不确定时求助”或“失败时降级”
- 单次任务可接受的 token、工具调用和人工复核成本上限是多少

### 5.2 需求评审阶段要固化的内容

| 项目 | 说明 |
| --- | --- |
| 任务目标 | 用户最终希望 Agent 完成什么 |
| 成功标准 | 完成任务时必须满足的条件 |
| 失败标准 | 触发失败、拦截或人工接管的条件 |
| 输入边界 | 允许处理的资料、格式、时长、轮次 |
| 输出边界 | 可以给出的结论、动作、写入范围 |
| 风险等级 | P0/P1/P2，按业务损失和错误可恢复性划分 |
| 成本预算 | 单任务 token、工具、人工复核预算 |
| 降级策略 | 失败时提示、重试、转人工或只读返回 |

### 5.3 是否适合做 Agent 的判断

优先考虑 Agent 的情况：

- 任务目标明确，但输入表达多样
- 需要调用多个工具、跨多个步骤完成
- 允许在不确定时补充说明或请求人工确认
- 任务量足够大，自动化节省的人力能覆盖运行成本

不优先做 Agent 的情况：

- 规则边界稳定，普通工作流即可解决
- 结果要求完全确定，且每一步都可规则化
- 错误代价高，但目前没有可靠的拦截和回退机制

## 6. S1 方案设计与可测性

### 6.1 单 Agent 的最低可测性要求

- 记录系统提示词、用户输入、知识来源版本
- 记录每次工具调用的工具名、参数、返回值、耗时、重试次数
- 记录关键中间状态和最终状态
- 记录模型版本、提示词版本、工具版本、运行环境
- 支持导出单次运行轨迹，便于复现

### 6.2 Multi-Agent 额外要求

- 每个节点有明确输入输出 schema
- 每个节点对共享 state 的写入字段有约束
- Supervisor 或调度器决策可追踪
- 节点间交接失败时能定位是哪一跳出错
- 支持 state 快照，便于做跨节点对账

### 6.3 测试侧在设计阶段要卡住的 8 项

- 是否存在可枚举的成功终态和失败终态
- 是否能复现外部工具的成功、空数据、超时和脏数据返回
- 是否能区分模型判断错误、工具选错、工具返回脏数据、状态污染
- 是否有版本号能关联到样本结果
- 是否允许对高风险动作加只读模拟或沙箱执行
- 是否能记录超长上下文截断位置
- 是否支持灰度开关和人工接管
- 是否预留线上坏例入库字段

## 7. S2 Eval 样本建设

### 7.1 样本来源

| 来源 | 作用 |
| --- | --- |
| 设计构造 | 覆盖标准流程和明确边界 |
| 线上翻车 | 覆盖已经发生过的问题 |
| 线上发现 | 覆盖高追问、高中断、高接管场景 |
| 故障注入 | 主动制造脏数据、超时、空值、状态污染 |

### 7.2 样本类型

| 类型 | 关注点 |
| --- | --- |
| 正常 | 标准任务是否稳定完成 |
| 边缘 | 模糊表达、缺信息、长上下文、权限边界 |
| 混沌 | 工具失败、空结果、脏数据、提示注入 |
| 记忆 | 多轮上下文、前后约束传递、状态污染 |

### 7.3 每条样本最少字段

| 字段 | 说明 |
| --- | --- |
| case_id | 样本唯一标识 |
| 业务场景 | 问答、诊断、规划、写入、审批等 |
| 风险等级 | P0/P1/P2 |
| 来源 | 设计构造、线上翻车、线上发现、故障注入 |
| 类型 | 正常、边缘、混沌、记忆 |
| 输入 | 用户输入、附件、上下文、环境前置状态 |
| 预期行为 | 允许的行为范围 |
| 禁止行为 | 不得出现的错误、越权、编造 |
| 评分规则 | 对应使用哪些评分器 |
| 证据 | 日志、截图、工具返回、state 快照 |

### 7.4 样本池组织方式

- `smoke`：5 到 10 条，覆盖主流程和 P0 风险
- `standard`：20 到 50 条，覆盖主要业务路径和已知失败模式
- `full`：50 条以上，加入长尾、混沌、记忆、回归样本

起步阶段先把 `smoke` 和 `standard` 建起来。`full` 适合提测、灰度和版本回归。

## 8. S3 评分器设计

### 8.1 推荐评分器分层

| 评分器 | 判断内容 | 适用场景 |
| --- | --- | --- |
| code grader | 字段、格式、数值、关键词、结构 | 确定性检查 |
| state grader | 最终状态、副作用、写入结果 | 有环境终态的任务 |
| tool grader | 工具名、参数、顺序、必要调用、多余调用 | 工具型 Agent |
| llm judge | 语义完整性、结论是否有依据、是否满足行为约束 | 语义型输出 |
| human review | 高风险动作、模糊边界、评分器校准 | P0 场景和抽检 |

### 8.2 LLM 评分器编写要求

- 明确评估维度，不写“回答是否正确”这种宽泛指令
- 给出通过和不通过条件
- 强制结构化输出，输出格式优先用 JSON
- 把工具返回值、状态快照、知识来源一起传入
- 先用明显错误样本校验评分器能拦下来，再投入批量运行

### 8.3 评分器组合顺序

执行顺序如下：

1. 先跑 code grader 和 state grader，快速拦截明显失败
2. 再跑 tool grader，确认过程是否越界或缺步
3. 最后只对需要语义判断的样本跑 llm judge
4. 对 P0 和争议样本做人审抽检

这样可以控制 token 成本，也能减少对 LLM 评分器的误信。

## 9. S4 分层执行

### 9.1 单 Agent 分层

| 层级 | 目标 | 示例 |
| --- | --- | --- |
| L0 依赖检查 | 模型、工具、权限、文件、接口可用 | 工具可连通、目录可写 |
| L1 单能力 | 单工具、单步骤能力可用 | 查日志、读文件、提取字段 |
| L2 单任务 | 一个完整任务能独立完成 | 读 PRD 后生成测试计划 |
| L3 业务 E2E | 真实业务链路可跑通 | 读取需求、识别风险、输出文档 |
| L4 混沌与回归 | 异常、边缘、长上下文、攻击场景可控 | 工具超时、注入、空值、重试 |

### 9.2 Multi-Agent 分层

| 层级 | 目标 | 重点 |
| --- | --- | --- |
| M1 节点级 | 单节点行为符合本节点约束 | 输入输出契约 |
| M2 交接级 | 上下游节点能正确消费彼此输出 | schema、state 字段 |
| M3 调度级 | Supervisor 或调度逻辑正确 | 节点激活顺序、重试与回退 |
| M4 全链路级 | 多节点整体完成任务 | 端到端成功率 |
| M5 一致性级 | 关键约束在多节点间不丢失 | 范围约束、权限约束、过滤条件 |

### 9.3 故障定位框架

发现问题后，按下面 4 层拆解：

- 推理层：前提理解错、结论跳步、看起来合理但依据错误
- 工具选择层：选错工具、参数错、时区错、顺序错
- API 执行层：接口空数据、脏数据、超时、重试不当
- 状态记忆层：上下文截断、共享 state 污染、约束丢失

## 10. S5 准入评审

### 10.1 准入维度

| 维度 | 问题 |
| --- | --- |
| 业务效果 | 任务是否完成，是否解决真实问题 |
| 风险控制 | 是否存在编造、越权、错误写入、错误结论 |
| 稳定性 | 重复执行是否稳定，异常时是否能恢复 |
| 可解释性 | 是否能定位失败步骤和证据 |
| 成本 | 单任务 token、工具、人工复核是否可接受 |

### 10.2 准入规则

- 出现任一 P0 失败样本，不进入生产候选
- `smoke` 样本必须全部通过
- `standard` 样本通过率、成本和人工接管率达到试点目标后，才进入灰度
- LLM 评分器与人工抽检偏差过大时，先校准评分器，不直接给版本结论
- 高风险写入类动作必须验证回退或人工确认机制

## 11. S6 上线运行与复盘

### 11.1 上线后要盯的指标

- 用户追问率
- 人工接管率
- 失败重试率
- 工具调用异常率
- 空数据或脏数据命中率
- 单任务平均 token 成本
- 单任务尾部成本，关注极端长上下文样本

### 11.2 坏例回流机制

每个线上问题都要至少回流 4 个动作：

1. 记录原始输入、环境状态、输出结果、工具轨迹
2. 标注故障层级和业务影响
3. 修复后补进 Eval 样本池
4. 在下一轮回归中固定复跑

### 11.3 周期复盘关注点

- 新问题是否集中在某类任务或某个节点
- 是否是样本覆盖不够，还是评分器不准
- Token 成本上涨来自上下文变长、重试增多，还是评分器使用过多
- 哪些样本长期没有发现问题，可以降级为抽检
- 哪些问题已多次出现，需要从架构或产品边界上收口

## 12. Token 与 ROI 口径

### 12.1 成本口径

至少记录 5 类成本：

- 候选 Agent 运行 token 成本
- 评分器 token 成本
- 工具/API 调用成本
- 人工抽检和复核成本
- 失败重跑和返工成本

### 12.2 基础公式

```text
单任务评测成本 = 候选 Agent token 成本 + 评分器 token 成本 + 工具/API 成本 + 人工复核成本

单轮版本回归成本 = 样本数 × 单任务评测成本 + 抽检成本

试点净收益 = 节省人工处理成本 - 评测运行成本 - 线上失败成本 - 返工成本
```

### 12.3 ROI 判断

继续投入的场景：

- 人工任务量大、重复度高
- 失败后可快速纠正，不会直接造成高损失
- 经过样本回归后，成功率和接管率在持续改善
- 每轮评测新增成本低于新增业务收益

收缩范围的场景：

- P0 风险高，但没有可靠拦截和回退
- 线上坏例持续集中在同一根因，短期无法修复
- 为了维持效果而不断拉长上下文，token 成本持续失控
- 评分器成本已经接近或高于人工直接处理成本

### 12.4 Token 控制

- 先跑 `smoke`，只有通过后再跑 `standard` 或 `full`
- 能用代码评分器的维度，不交给 LLM 评分器
- 把长文档拆成可定位片段，避免每次整包送入上下文
- 对工具返回做快照复用，减少重复调用
- 只对有争议或高风险样本启用高成本评分器
- 对长上下文样本单独设预算上限，超限直接标记失败或转人工

## 13. 角色分工

| 角色 | 主要责任 |
| --- | --- |
| 产品 | 定义任务目标、成功标准、边界和业务价值 |
| 测试/质量 | 设计样本、评分器、分层执行和复盘机制 |
| 工程 | 提供日志、状态、工具契约、灰度和回退能力 |
| 算法/模型 | 优化提示词、路由、工具策略和评估器提示词 |
| 业务方 | 参与坏例确认、人工抽检和价值判断 |
