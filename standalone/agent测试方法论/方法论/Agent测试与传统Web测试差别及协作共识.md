# Agent测试与传统Web测试差别及协作共识

## 1. 使用目的

传统 Web 项目里，产品、测试、开发通常默认按同一套测试前提协作：

- 需求是相对确定的
- 输入输出路径可以枚举
- 功能点、接口、页面状态可以做明确断言
- 主要问题集中在代码实现、接口契约、前后端联调

Agent 项目不是这套前提。它的任务目标、过程轨迹、工具调用、上下文状态、模型判断和 token 成本都会直接影响质量结果。如果团队还按传统 Web 项目的协作方式推进，常见结果是：

- 产品按“功能点列表”提需求，但没有定义失败边界和降级策略
- 测试按“页面流程 + 接口断言”写用例，但抓不住推理错误和状态污染
- 开发按“功能完成”交付，但没有补齐轨迹日志、状态快照和工具证据

本文件用于在需求评审、项目启动和阶段复盘时直接对齐产品、测试、开发三方职责。

## 2. 与传统 Web 测试的核心差别

| 维度 | 传统 Web 项目测试 | Agent 项目测试 |
| --- | --- | --- |
| 需求对象 | 功能、页面、接口、状态流转 | 任务目标、行为约束、工具策略、失败边界 |
| 测试对象 | 确定性逻辑 | 带不确定性的任务执行系统 |
| 成功判断 | 页面正确、接口正确、数据正确 | 任务完成、过程可控、状态正确、成本可接受 |
| 主要断言方式 | UI/接口/数据库精确断言 | 代码评分器 + 状态评分器 + 工具评分器 + LLM评分器 |
| 风险类型 | 功能缺失、接口错误、兼容性、性能问题 | 推理错误、工具误用、脏数据误判、记忆污染、幻觉、越权 |
| 测试样本来源 | 需求设计、接口文档、页面流程 | 真实用户请求、线上翻车、故障注入、多轮上下文 |
| 可观测性要求 | 接口日志、前端埋点、错误堆栈 | 输入、提示词、工具参数、工具返回、状态快照、节点轨迹 |
| 准入依据 | 功能通过率、缺陷等级、回归结果 | 业务效果、风险控制、稳定性、token成本、人工接管率 |
| 线上处理方式 | 缺陷修复和回归 | 坏例回流、样本入库、评分器校准、成本漂移监控 |

## 3. 不能直接套用传统 Web 测试方法的原因

### 3.1 需求不是“功能枚举”就够了

传统 Web 项目里，产品写清页面、按钮、接口规则，测试基本可以据此展开。Agent 项目还需要额外回答：

- 什么情况下允许 Agent 回答“不确定”
- 什么情况下必须转人工
- 什么工具可以调用，什么工具不能调用
- 什么错误可以接受，什么错误必须拦截
- 单次任务成本上限是多少

这些如果不在需求阶段讲清楚，测试和开发后面只能各自猜。

### 3.2 输出不能只做固定断言

传统 Web 项目常见断言是：

- 接口返回码等于 200
- 页面显示字段 A/B/C
- 数据库中新增一条记录

Agent 项目里，最终回答即使“看起来没问题”，也可能：

- 结论依据错了
- 用错了工具
- 漏掉关键约束
- 在空数据上给出确定性结论

所以测试重点要从“结果是否等于预期”扩展到“行为是否在约束内”。

### 3.3 线上问题不只来自代码实现

传统 Web 项目大多数问题最终都能落回代码、接口或配置。Agent 项目里，失败可能来自：

- 模型推理把前提理解错
- 工具参数解析错
- 外部接口返回脏数据
- 多轮上下文把早期限制条件冲掉

这决定了测试、开发、产品三方必须一起看问题，而不是默认“这是开发 bug”。

## 4. 产品、测试、开发分别要改什么

## 4.1 产品

传统 Web 项目里，产品更关注：

- 用户流程
- 页面交互
- 字段和规则
- 验收条件

Agent 项目里，产品还需要额外定义：

- 任务目标，而不是只列功能动作
- 成功边界和失败边界
- 允许的自动化范围和必须人工确认的范围
- 风险等级，尤其是误答、误判、误操作的业务影响
- 成本约束，包括 token、工具调用、人工复核预算

产品需要补充的内容：

- 需求评审时增加“失败场景”和“降级策略”章节
- 不用“希望尽量回答准确”这类泛化描述，改成可判断标准
- 对高风险任务先收小范围，不要一开始就放全量场景

## 4.2 测试

传统 Web 项目里，测试的主线通常是：

- 功能测试
- 接口测试
- 回归测试
- 兼容性和性能测试

Agent 项目里，测试要额外承担：

- 样本建设，而不是只写用例
- 评分器设计，而不是只做人工判定
- 坏例回流，而不是只提 bug
- 成本监控，而不是只看通过率

测试需要补充的内容：

- 样本优先从真实输入和线上坏例来建，不先追求数量
- 对高风险任务先定义禁止行为，再定义理想行为
- 分开判断推理层、工具层、API 层、状态层问题
- 把 LLM 评分器当补充，不替代代码和状态校验

## 4.3 开发

传统 Web 项目里，开发交付重点通常是：

- 页面和接口功能完成
- 异常处理完成
- 日志和监控基本可查

Agent 项目里，开发还要额外交付：

- 可追踪的工具调用日志
- 可回放的输入、输出和状态快照
- 节点间 schema 和共享 state 约束
- 高风险动作的模拟执行、回退和人工确认能力

开发需要补充的内容：

- 把可测性字段当正式交付项，不是联调时临时补
- 对写入型工具、外部依赖、长上下文任务优先做防护
- 不只交“功能跑通”，还要交“失败后能定位”

## 5. 团队需要确认的 8 条事项

### 事项 1

Agent 项目的需求评审，不等于把传统 PRD 再讲一遍。

必须额外讲清：

- 自动化范围
- 人工接管点
- 高风险误判场景
- 成本预算

### 事项 2

Agent 测试的对象不是“页面 + 接口”，而是“任务 + 过程 + 状态 + 成本”。

### 事项 3

能不能上线，不只看功能有没有跑通，还要看：

- 是否出现 P0 风险
- 是否能解释失败
- 是否能承受运行成本

### 事项 4

样本比用例更重要。尤其是线上坏例、用户追问、故障注入样本。

### 事项 5

评分器是正式资产，不是测试临时脚本。代码评分器、状态评分器、工具评分器、LLM评分器都要版本化。

### 事项 6

没有轨迹、状态和工具证据，就没有办法稳定评测 Agent。

### 事项 7

线上问题不默认归开发，也不默认归模型。先按推理、工具、API、状态四层拆解。

### 事项 8

Agent 项目必须把 token 成本和人工接管率纳入版本结论。质量和成本是同一份结论里的两个字段。

## 6. 协作方式

## 6.1 需求评审阶段

产品、测试、开发至少对齐下面这些问题：

| 问题 | 产品 | 测试 | 开发 |
| --- | --- | --- | --- |
| 任务目标是什么 | 负责定义 | 负责确认可测 | 负责确认可实现 |
| 失败边界是什么 | 负责定义 | 负责转成样本 | 负责落成保护逻辑 |
| 哪些动作高风险 | 负责判业务影响 | 负责纳入 P0 样本 | 负责增加防护和回退 |
| 单次成本上限是多少 | 负责给成本边界 | 负责评测监控 | 负责技术优化 |

## 6.2 联调阶段

不要只看“跑通没有”，要同步看：

- 轨迹有没有记录全
- 工具参数能不能查
- state 快照能不能导出
- 失败样本能不能复现

## 6.3 回归和准入阶段

回归和准入阶段统一看一张表：

- 主流程通过率
- P0 失败数
- 主要失败层级分布
- 平均 token 成本
- 人工接管率
- 本轮新增坏例数

## 7. 哪些项目更像传统 Web，哪些项目必须按 Agent 测试来做

更像传统 Web 的情况：

- 规则稳定
- 输入结构固定
- 输出要求完全确定
- 基本不依赖推理和多工具协同

必须按 Agent 测试来做的情况：

- 输入自然语言表达差异大
- 结果依赖推理、检索、规划或工具组合
- 存在多轮状态和上下文记忆
- 错误会直接影响业务判断或写入动作

## 8. 项目启动时直接确认的 10 个问题

1. 这个任务为什么一定要用 Agent，而不是规则流或普通页面流程？
2. 如果 Agent 判断错了，业务影响是什么？
3. 什么情况下允许它回答“不确定”？
4. 什么情况下必须转人工？
5. 哪些工具允许调，哪些工具禁止调？
6. 哪些状态必须记录下来？
7. 当前最怕的 3 个线上翻车场景是什么？
8. 单次任务的 token 成本预算是多少？
9. 失败样本谁来确认、谁来入库、谁来复测？
10. 哪个指标会直接影响下一阶段推进决策？

## 9. 结论

传统 Web 项目测试，主要验证功能和实现是否正确。  
Agent 项目测试，主要验证任务是否完成、行为是否受控、失败是否可收敛、成本是否可接受。
