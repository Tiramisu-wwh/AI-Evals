# 专业词汇附录

本附录用于解释这组 Agent 测试文档里反复出现的专业词，方便产品、测试、开发在评审时统一理解。

## 1. 常用词汇

| 词汇 | 说明 | 在当前文档里的用法 | 重点关注角色 |
| --- | --- | --- | --- |
| Agent | 能根据任务目标自己规划步骤、调用工具、输出结果的系统，不只是单次问答模型。 | 用来指代带推理、带工具、带状态的任务执行系统。 | 产品、测试、开发 |
| 传统 Web 项目测试 | 面向页面、接口、数据库和确定性逻辑的测试方式。 | 用来做对比，说明 Agent 测试和普通功能测试不同。 | 产品、测试、开发 |
| 任务目标 | 用户最终希望 Agent 完成的事情。 | 例如“定位故障原因”“生成测试计划”“汇总问题单”。 | 产品、测试 |
| 成功终态 | 一次任务完成后必须达到的结果状态。 | 用来判断任务是否真正完成，不只看模型回复。 | 产品、测试、开发 |
| 失败终态 | 一次任务被判定为失败时的结果状态。 | 例如“未生成文件”“错误写入”“没有证据却给确定性结论”。 | 产品、测试、开发 |
| 行为约束 | Agent 可以做什么、不能做什么。 | 用来限制工具调用、输出方式、写入动作和权限范围。 | 产品、测试、开发 |
| 可观测性 | 出问题后，能不能从日志、轨迹、状态里看出它做了什么。 | 主要看输入、工具参数、工具返回、状态快照、耗时等证据。 | 测试、开发 |
| 轨迹 | Agent 一次任务执行过程的记录。 | 包括它先做了什么、后做了什么、调了哪些工具。 | 测试、开发 |
| 工具调用 | Agent 为完成任务而去调外部能力。 | 例如查知识库、读文件、调接口、写文件。 | 测试、开发 |
| 工具误用 | Agent 调了不该调的工具，或工具参数、顺序不对。 | 这是 Agent 测试里的高频问题之一。 | 测试、开发 |
| 工具参数 | 调工具时传入的参数。 | 例如时间范围、用户 ID、文件路径、环境名称。 | 测试、开发 |
| 状态 | Agent 在执行过程中的中间信息或最终信息。 | 例如“当前已查到的结果”“当前目标环境”“当前待处理任务列表”。 | 测试、开发 |
| 共享 state | 多个 Agent 或多个节点共用的一份状态数据。 | 上游节点写入，下游节点读取。如果写错或丢字段，后续节点都会受影响。 | 测试、开发 |
| state 快照 | 某个时刻的状态记录。 | 用来排查任务在哪一步把状态写坏了。 | 测试、开发 |
| Multi-Agent | 由多个 Agent 或多个节点协同完成一个任务。 | 常见于“分析 -> 规划 -> 执行 -> 审核”这类流程。 | 产品、测试、开发 |
| 节点 | Multi-Agent 流程中的一个处理环节。 | 例如需求分析节点、工具执行节点、结果审核节点。 | 测试、开发 |
| Supervisor | 负责决定当前应该由哪个节点继续执行的调度角色。 | 主要用于 Multi-Agent 场景。 | 测试、开发 |
| schema | 字段结构约定。 | 用来说明输入、输出、状态里应该有哪些字段、字段类型是什么。 | 测试、开发 |
| Eval | 一组可重复执行的评测样本和判定规则。 | 用来持续验证 Agent 版本效果，不是一次性手工测试。 | 产品、测试、开发 |
| 样本 | Eval 里的单条测试任务。 | 包括用户输入、前置状态、预期行为、禁止行为、评分方式。 | 测试 |
| 坏例 | 已经暴露出问题的真实案例。 | 通常来自线上翻车、用户追问、回归失败。修复后应进入样本池反复复测。 | 产品、测试、开发 |
| 线上翻车 | 真实使用过程中出现明显错误。 | 例如误判、答非所问、错用工具、错误写入。 | 产品、测试、开发 |
| 故障注入 | 人为制造异常条件来测系统。 | 例如空数据、脏数据、超时、权限不足、提示注入。 | 测试、开发 |
| 提示注入 | 输入内容里夹带“忽略之前要求”“直接输出成功”之类干扰指令。 | 用来测试 Agent 会不会被恶意内容带偏。 | 测试、开发 |
| 评分器 | 用来判断样本通过或失败的检查逻辑。 | 分代码评分器、状态评分器、工具评分器、LLM 评分器。 | 测试、开发 |
| 代码评分器 | 用规则或脚本做确定性检查。 | 适合查字段、格式、数值、是否生成文件。 | 测试、开发 |
| 状态评分器 | 根据最终状态或副作用判断结果。 | 适合查环境终态、写入结果、任务副作用。 | 测试、开发 |
| 工具评分器 | 专门检查工具有没有被正确使用。 | 看工具名、参数、顺序、必要调用、多余调用。 | 测试、开发 |
| LLM 评分器 | 用另一个模型判断语义层面的通过或失败。 | 适合看回答是否满足约束、结论是否有依据。 | 测试、开发 |
| 人工复核 | 由人来确认结果。 | 适合高风险场景、争议样本、评分器校准。 | 产品、测试、业务 |
| smoke | 最小回归集。 | 一般 5 到 10 条，先挡住主流程和 P0 风险。 | 测试 |
| standard | 常规回归集。 | 在 smoke 基础上加主要业务路径和已知失败模式。 | 测试 |
| full | 全量回归集。 | 用于灰度前、版本回归或专项检查。 | 测试 |
| P0/P1/P2 | 风险或问题优先级。 | P0 一般指必须拦截的问题，P1/P2 风险依次降低。 | 产品、测试、开发 |
| 推理层问题 | Agent 理解前提或做判断时出错。 | 例如把“数据缺失”理解成“指标归零”。 | 测试、开发 |
| API 层问题 | 外部接口返回异常或数据质量差。 | 例如空数组、脏数据、超时、字段缺失。 | 测试、开发 |
| 状态污染 | 旧信息、错信息进入后续步骤，影响后面的判断。 | 常见于长对话或共享 state 场景。 | 测试、开发 |
| 人工接管 | 系统不继续自动执行，转给人工处理。 | 用于高风险动作、低置信度结果、超预算场景。 | 产品、测试、开发 |
| 降级 | 系统改用更保守的处理方式。 | 例如只返回参考信息、不执行写入、提示人工确认。 | 产品、测试、开发 |
| token 成本 | 模型处理一次任务消耗的 token 对应成本。 | 要和效果、人工接管率一起看。 | 产品、测试、开发 |
| ROI | 投入产出比。 | 常用来判断 Agent 方案是否值得继续投资源。 | 产品、管理、开发 |

## 2. 几个容易混淆的词

### 2.1 用例 和 样本

- 用例：传统测试常用写法，偏功能分支和页面流程。
- 样本：Agent 评测常用写法，偏真实输入、真实上下文和真实失败场景。

这组文档里更常用“样本”，因为 Agent 测试更依赖真实输入和坏例回放。

### 2.2 日志、轨迹、state 快照

- 日志：系统运行时写下来的记录，范围最广。
- 轨迹：一次任务从开始到结束的动作顺序。
- state 快照：某个时间点的状态切片。

排查问题时，通常要三者一起看。

### 2.3 错误回答、错误执行、错误状态

- 错误回答：最后说错了。
- 错误执行：工具调错了、步骤顺序错了。
- 错误状态：中间状态被写坏了，后面都跟着错。

传统 Web 测试更容易盯住“结果错了”，Agent 测试要把这三类拆开看。

## 3. 使用说明

- 产品不确定某个词时，先看“说明”和“在当前文档里的用法”两列。
- 测试写样本、评分器、复盘时，优先统一“坏例”“评分器”“状态污染”“故障注入”这几个词。
- 开发评审方案时，优先统一“共享 state”“schema”“轨迹”“state 快照”“人工接管”这几个词。
