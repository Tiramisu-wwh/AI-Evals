# AI评测接入指南

## 1. 指南用途

这份指南用于说明已有系统如何接入 AI 评测体系，重点回答下面两个问题：

- 公司里已经有一个知识库，怎么快速接入这套 AI 评测体系？
- 公司里已经有一个 Agent，怎么快速接入这套 AI 评测体系？

总原则：`不要求先重做系统，只需要补一层轻量接入适配，就能挂上统一的样本、评分器和复盘机制。`

## 2. 接入后的实际形态

一个系统接入 AI 评测之后，最终会多出 5 个东西：

1. `成功标准`
2. `黄金样本集`
3. `运行入口`
4. `评分器组合`
5. `结果报告与复盘`

这 5 个东西里，只有“运行入口”需要贴近现有系统，其余 4 个都可以复用这套文档包里的标准和模板。

## 3. 接入适配层说明

接入适配层用于完成下面几类映射：

- 把系统的输入映射成统一任务格式
- 把系统的输出映射成统一结果格式
- 如果拿得到检索结果、引用、轨迹或终态，就一并挂上来
- 让评分器和模板都能读懂

接入时不要只问“系统接口怎么调”，还要同步确定 Eval 数据来源。首批样本可以少，但必须尽量真实；每次线上问题修复后，都要判断是否新增为 Eval 或回归样本。

## 4. 三个接入等级

### L0：黑盒接入

只拿得到：

- 输入
- 输出

典型情况：

- 现阶段只有 UI 或 API 返回答案
- 拿不到检索结果和内部日志

### L1：灰盒接入

在 L0 基础上，再补至少一种中间信息：

- 检索结果
- 引用文档
- 工具调用日志
- 中间状态

典型情况：

- 已经能拿到部分链路数据
- 希望开始做问题归因

### L2：深度接入

具备结构化链路数据：

- 全量轨迹
- 环境终态
- 批跑脚本
- 结果落库或看板

典型情况：

- 已验证试点有效
- 准备进入半自动化

## 5. 知识库快速接入

### 5.1 最小接入要求

如果公司里已经有一个知识库问答系统，最小只要准备 4 项：

1. 一批真实问题，建议先 30 到 50 条
2. 每题对应的正确依据或证据文档
3. 一个运行入口，可以是 API、脚本，甚至人工导出
4. 最好能拿到引用文档或检索结果

如果暂时拿不到第 4 项，也可以先按 L0 黑盒接入。

### 5.2 接入步骤

1. 用 [成功标准模板](../templates/成功标准模板.md) 定义这类知识问答什么叫“答得好”。
2. 用 [黄金样本模板](../templates/黄金样本模板.md) 建立首批问题集。
3. 选择 [RAG/问答操作手册](../playbooks/RAG问答操作手册.md) 作为主手册。
4. 用 [评分器模板](../templates/评分器模板.md) 定义规则检查和 LLM judge。
5. 先跑一轮小样本 dry-run。
6. 用 [评测结果模板](../templates/评测结果模板.md) 和 [复盘模板](../templates/复盘模板.md) 出结果并开复盘会。

### 5.3 知识库快速接入示例

#### 场景

公司已经有一个“制度流程知识库”，员工可以输入问题，系统返回答案和引用文档。

#### 最小输入

- 40 条真实问题
- 每条问题对应的制度文档编号
- 一个可调用接口：输入问题，输出答案和引用文档列表
- 每条样本标记来源，例如线上翻车、线上发现、设计构造

#### 接入形态

- 能力类型：`RAG/问答`
- 接入等级：`L1 灰盒接入`
- 主操作手册：`RAG/问答操作手册`
- 首批评分维度：相关性、忠实度、完整性、清晰度、边界控制

#### 第1周就能产出的东西

- 一份成功标准
- 一份 40 条黄金样本集
- 一套基础评分器
- 一份首轮评测报告

#### 第1轮典型输出

- Top failure mode：`retrieval_miss`
- 代表样本：请假审批流程相关问题引用了旧制度
- 行动项：更新知识源、修正检索排序、补充回归样本

## 6. Agent 快速接入

### 6.1 最小接入要求

如果公司里已经有一个 Agent，最小建议准备 4 项：

1. 一组任务清单
2. 一个运行入口
3. 最终结果或环境终态
4. 最好能拿到工具调用日志或执行轨迹

如果暂时没有轨迹，也可以先按 L0 或 L1 做结果导向接入。

### 6.2 接入步骤

1. 用 [成功标准模板](../templates/成功标准模板.md) 定义什么叫“任务完成”。
2. 用 [黄金样本模板](../templates/黄金样本模板.md) 建立任务集。
3. 选择 [Agent工作流操作手册](../playbooks/Agent工作流操作手册.md) 作为主手册。
4. 用 [评分器模板](../templates/评分器模板.md) 定义终态检查、关键工具检查和轨迹抽检规则。
5. 先跑一轮以“结果和终态”为主的评测。
6. 再逐步补轨迹归因和回归集。

### 6.3 Agent 快速接入示例

#### 场景

公司已经有一个“工单处理 Agent”，会读取工单、查询知识库、给出处理建议，并更新工单状态。

#### 最小输入

- 25 个典型工单任务
- 每个任务对应的预期终态，例如“已分派”“已升级”“已关闭”
- 一个执行入口，可以触发 Agent 跑一遍
- 一份工具调用日志，至少包含查询知识库和更新工单状态
- 每个任务标记案例类型，例如正常、边缘、混沌、记忆或上下文

#### 接入形态

- 能力类型：`Agent 工作流`
- 接入等级：`L1 灰盒接入`
- 主操作手册：`Agent 工作流操作手册`
- 首批评分维度：任务成功率、状态正确率、工具调用有效性、异常恢复能力

#### 第1周就能产出的东西

- 一份任务成功标准
- 一份 25 条任务集
- 一套终态检查规则
- 一份首轮评测报告

#### 第1轮典型输出

- Top failure mode：知识库查询正确，但工单没有更新到正确状态
- 根因初判：工具调用参数错误
- 行动项：修 Agent 的状态映射逻辑，并把该任务加入回归集

## 7. 接入谁来负责

默认分工如下：

- 测试/质量：牵头接入、组织模板、收口字段和复盘
- 产品：定义成功标准和优先级
- 工程/算法：提供运行入口、日志、结果字段
- 业务专家：提供样本真相和边界判断

## 8. 接入完成的判断标准

一个系统不需要做到平台化，满足以下条件就算“完成接入”：

- 已经选定对应能力类型和主操作手册
- 已经有首批成功标准
- 已经有首批黄金样本
- 每条样本都有输入/任务、预期行为和可操作评估标准
- 已经能跑出一轮评测结果
- 已经能开一轮复盘会并形成行动项

满足以上条件后，可视为完成首轮接入。
