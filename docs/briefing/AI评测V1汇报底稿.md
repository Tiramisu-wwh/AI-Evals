# AI 评测 V1 汇报底稿

## 1. 背景

当前团队已经准备推进 AI 评测体系建设，但现状通常会遇到三类问题：

- AI 功能“感觉不稳定”，但没有统一办法证明哪里变好了、哪里退步了
- 产品、工程、测试/质量对“什么叫答得好”理解不一致
- 出现问题后只能靠个案排查，难以沉淀成复用标准

V1 当前优先建设执行标准、模板和运行节奏。

## 2. V1 的定位与范围

V1 用于试点阶段的评估、复盘和优化协同，当前范围不包含发布门禁。

当前建设内容包括：

- 统一术语
- 统一模板
- 统一运行闭环
- 统一复盘与归因方式

当前阶段范围外内容：

- 全自动门禁
- 全业务平台化
- 大而全的数据治理

## 3. 为什么由测试/质量牵头

测试/质量适合作为第一责任主体，主要原因是它天然承担三类工作：

- 把模糊体验转成结构化检查项
- 组织跨角色对齐和复盘
- 把单点问题沉淀成可复用标准

在 V1 中：

- 产品负责定义成功标准和业务优先级
- 工程/算法负责实现修复和埋点
- 业务专家负责提供真相和校准判断
- 测试/质量负责跑机制

## 4. V1 交付物

第一版交付物采用 `文档 + 模板包 + 汇报材料` 的组合：

- 一份总框架
- 三份操作手册
- 一份内部知识问答试点包
- 一组模板
- 一份管理层与执行团队都能复用的汇报底稿
- 一份自动化与智能化待办清单

## 5. 为什么先从 RAG/内部知识问答试点

三个原因：

1. 最容易界定业务边界和证据来源
2. 最容易建立黄金样本和 judge 校准机制
3. 最容易先跑出一套可复用的周节奏

V1 仍然会覆盖：

- RAG/问答
- Agent 工作流
- 内容生成

但第一批试点只把 RAG 写得最细、跑得最深。

## 6. 推进重点

- 定总框架，明确边界、术语、职责和运行闭环
- 建模板和首批样本，形成可执行的基础材料
- 跑一轮完整评测，输出结果报告和复盘结论
- 推动一轮修复和复测，确认这套机制能够支撑优化闭环

## 7. 如何衡量 V1 是否成功

当前优先看机制是否跑通：

- 是否已经说清什么叫答得好
- 是否已经有结构化黄金集
- 是否已经能稳定跑一轮评测
- 是否已经识别出稳定的 failure modes
- 是否已经推动至少一次明确优化闭环

## 8. 后续演进

### 阶段 1：手工化

- 人工维护样本
- 脚本或手工跑评测
- 人工复盘

### 阶段 2：半自动化

- 样本库结构化
- grader 批量执行
- 周报自动生成

### 阶段 3：智能化

- 自动聚类 failure modes
- 自动发现分数漂移
- 自动推荐根因与回归集

## 9. 当前需要管理层支持的事项

- 明确试点业务域
- 确认可投入的产品、工程、业务专家资源
- 接受 V1 先服务试点优化，不直接承担上线门禁
- 支持第二阶段的半自动化建设投入
