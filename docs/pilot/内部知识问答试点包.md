# 内部知识问答试点包

## 1. 试点目标

本试点用于验证 AI Evals V1 能否在一个真实业务边界内稳定运行，并完成至少一轮“发现问题、推动改进、再次验证”的闭环。

试点需要回答 4 个问题：

1. 什么叫答得好？
2. 当前主要失败模式是什么？
3. 评测结果能否稳定复现这些问题？
4. 结果能否指导一次明确优化？

## 2. 试点范围

### 推荐知识域

从以下类别中选择一个作为首个试点：

- 制度流程
- 产品说明
- 内部 SOP

### 知识域选择标准

- 边界清楚，能说清“不属于这个试点的内容”
- 有可靠知识源作为证据基础
- 真实用户问题足够多
- 出错成本可控，但有明确业务价值

## 3. 试点成功标准

### 机制成功

- 样本集结构稳定
- judge 与人工有可接受的一致性
- 质量团队能独立组织周节奏

### 业务成功

- 至少识别出 3 类稳定 failure modes
- 至少推动 1 次明确改进
- 改进后在下一轮评测中得到验证

## 4. 样本集设计

### 初始规模

- 建议建立 `50-100 条` 初始黄金样本

### 配比建议

| 类型 | 数量建议 | 说明 |
| --- | --- | --- |
| 高频常见题 | 20-40 | 真实用户最常问 |
| 难题 | 10-20 | 需要更深上下文理解 |
| 边界题 | 10-20 | 歧义、上下文不足、跨范围 |
| 拒答/澄清题 | 10-20 | 不应直接回答的场景 |

### 每条样本必须记录

- 用户问题
- 预期答案或关键点
- 证据来源
- 相关文档链接/编号
- 难度
- failure mode 标签
- 当前状态

## 5. 试点 grader 方案

### 5.1 Code grader

第一版建议覆盖：

- 输出格式检查
- 引用存在性检查
- 空答检查
- 越权引用检查
- 基础安全和明显错误检查

### 5.2 LLM judge

分维度评分：

- 相关性
- 忠实度
- 完整性
- 清晰度

每个维度输出：

- 分数
- 简短理由
- 是否建议人工复核

### 5.3 人工抽检

抽检优先级：

1. 高风险问题
2. judge 标记为需复核的问题
3. 边界样本
4. 新增 failure mode 代表样本

## 6. 失败模式标签

本试点固定使用以下一级标签：

- `knowledge_missing`
- `retrieval_miss`
- `retrieval_noise`
- `prompt_misaligned`
- `model_hallucination`
- `answer_incomplete`
- `answer_unclear`
- `boundary_failure`

## 7. 推进阶段

### 阶段 1：试点建模

- 冻结试点范围
- 定义成功标准
- 选定知识域
- 建立首批样本
- 编写模板和评分器初稿

### 阶段 2：校准

- 完成人工标注子集
- 校准 judge
- 做第一轮 dry-run
- 修订 rubric 和 failure mode 标签

### 阶段 3：试跑

- 跑一轮完整评测
- 生成结果报告
- 输出 Top failure modes
- 确认优先改进项

### 阶段 4：验证

- 跟踪修复
- 复跑评测
- 验证问题是否改善
- 形成试点总结

## 8. 评测结果输出要求

每轮至少输出以下 6 项：

1. 总分
2. 分维度分数
3. Top failure modes
4. 代表性失败样本
5. 根因归类
6. 下一步行动项

## 9. 试点会议机制

### 周复盘会

参与角色：

- 测试/质量
- 产品
- 工程/算法
- 业务专家

固定议程：

1. 回顾本周分数和变化
2. 审 3-5 个代表样本
3. 确认 failure modes 是否需要新增或合并
4. 确认 Top 问题和责任人
5. 冻结下轮验证点

## 10. 试点结束后的产物

- 试点总结文档
- 稳定样本集
- 稳定 grader 规范
- 复盘方法和会议机制
- 半自动化建设待办清单
