# AI Evals V1 总框架

## 1. 定位

AI Evals V1 是一套由 `测试/质量` 牵头运营的执行标准与执行流程，用于帮助团队在 AI 功能上线前和试点阶段：

- 明确什么叫“做得好”
- 把主观体验转成可复用的评测任务和评分规则
- 用一致的运行节奏发现问题、归因问题、推动改进
- 为后续半自动化和智能化建设预留统一字段和流程接口

V1 不追求平台完备，也不直接承担发布门禁。它的核心目标是先让团队跑通机制并证明可复用。

## 2. 适用边界

### V1 适用

- 新建或正在试点的 AI 产品能力
- 需要先统一成功标准和评测方法的场景
- 团队需要建立共同术语、模板和复盘节奏的场景
- 质量团队需要牵头组织 AI 质量评估的场景

### V1 不优先处理

- 完整的线上质量门禁系统
- 统一的评测平台与全自动流水线
- 跨全部业务域的大规模生产级数据集治理
- 所有模型和系统的一次性统一评估

## 3. Eval 与传统测试用例的区别

AI 评测不能简单理解成“更高级的测试用例”。传统测试用例通常从系统设计出发，枚举分支和边界；AI Eval 更强调从真实用户行为、线上翻车案例和业务风险出发，验证系统是否能在不确定输入下给出可接受的结果。

| 维度 | 传统测试用例 | AI Eval |
| --- | --- | --- |
| 起点 | 系统设计、功能分支 | 真实用户问题、线上失败、业务风险 |
| 输出判断 | 确定性断言 | 语义判断、质量评分、终态验证 |
| 覆盖目标 | 尽量覆盖分支 | 优先覆盖真实高价值问题 |
| 维护方式 | 随需求和代码变更维护 | 随线上反馈、失败模式和模型能力变化维护 |

V1 的默认原则是：`20 条真实翻车沉淀出来的 Eval，通常比 200 条凭空构造的标准问法更有价值`。因此，样本集建设要优先收集真实问题、线上失败、线上异常和故障注入案例，再补充设计构造样本。

## 4. 统一术语

| 术语 | 定义 | V1 中如何使用 |
| --- | --- | --- |
| `task` | 一条可评估任务或问题 | 最小评测单元 |
| `trial` | 对同一任务的一次执行 | 用于观察随机性与稳定性 |
| `grader` | 对输出、轨迹或终态进行评分的逻辑 | 分为 code、llm、human 三类 |
| `golden set` | 经过筛选和标注的黄金样本集 | 作为回归和对比基线 |
| `failure mode` | 一类可重复出现的问题模式 | 用于复盘和聚类归因 |
| `capability eval` | 衡量“当前能做到什么”的评测 | 用于探索能力上限 |
| `regression eval` | 衡量“原来能做到的是否仍然做到”的评测 | 用于防回退 |
| `eval source` | 样本来源 | 区分设计构造、线上翻车、线上发现、故障注入 |
| `case type` | 样本类型 | 区分正常、边缘、混沌、记忆/上下文场景 |

## 5. 评测组件模型

V1 推荐把任意评测都拆成 3 类组件：

| 组件 | 说明 | 对应产物 |
| --- | --- | --- |
| 评估数据集 | 一批可重复执行的任务、问题或场景 | 黄金样本集、回归集、badcase 集 |
| 评估指标 | 定义“什么叫好”的维度和阈值 | 成功标准、评分权重、准入线 |
| 评估器 | 根据指标执行打分或判断的逻辑 | 规则评估器、LLM 评估器、人工评审 |

每条 Eval 至少需要具备 3 个要素：

- `输入/任务`：用户问题、任务描述或场景触发条件。
- `预期行为`：标准答案、关键事实、终态要求或必须/禁止行为。
- `评估标准`：可操作、可判断的检查规则，不能只写“要准确”“要完整”。

## 6. 目标分层

| 目标层 | 关注问题 | 典型指标 |
| --- | --- | --- |
| 业务目标 | 这项 AI 能力是否推动业务结果 | 解决率、转化、人工替代率、时效 |
| 用户体验目标 | 用户是否感受到有帮助、可信、顺畅 | 满意度、帮助性、清晰度、可操作性 |
| 模型能力目标 | 模型或链路本身是否具备所需能力 | 相关性、忠实度、任务成功率、稳定性 |
| 质量与风险目标 | 是否满足基本安全、合规、边界控制 | 幻觉率、越权率、拒答正确率、安全命中率 |

V1 要求每个试点至少同时定义这 4 层中的前 3 层指标，避免只有技术分数没有业务价值，或只有业务指标没有诊断抓手。

## 7. 角色分工

| 角色 | 核心职责 | 在 V1 中的主输出 |
| --- | --- | --- |
| 测试/质量 | 牵头方法、组织运行、维护模板、推动复盘 | 样本集、grader 规范、结果报告、复盘纪要 |
| 产品 | 定义成功标准、排序失败优先级、判断价值 | 成功标准、用户场景、业务优先级 |
| 工程/算法 | 落地实现、修复问题、补埋点与自动化 | 修复方案、链路数据、执行脚本 |
| 业务专家 | 提供领域真相、参与标注、校准评分 | 证据来源、边界判断、人工校准 |

### 默认协作机制

- 测试/质量负责“机制是否稳定运行”
- 产品负责“评测是否对准正确问题”
- 工程/算法负责“问题是否被真实修复”
- 业务专家负责“评分是否接近真实业务判断”

## 8. 运行闭环

V1 固定使用以下闭环，不建议项目自定义流程名称或跳步骤：

1. `定义成功`
2. `建立黄金集`
3. `设计 grader`
4. `跑评测`
5. `复盘改进`

### 8.1 定义成功

输出：

- 业务目标
- 关键用户场景
- 评估维度和权重
- 及格线和优秀线

### 8.2 建立黄金集

输出：

- 任务/问题清单
- 参考答案或证据来源
- 难度标签
- 失败模式标签
- 样本来源和案例类型

### 8.3 设计 grader

输出：

- code grader 规则
- llm judge rubric
- 人工抽检策略
- 通过条件

### 8.4 跑评测

输出：

- 总分和分维度分数
- 代表性样本
- 趋势变化
- 问题清单

### 8.5 复盘改进

输出：

- Top failure modes
- 根因归类
- 改进行动
- 责任人和下一轮验证点

## 9. 周运行节奏

V1 推荐按周循环运行，便于质量团队组织协作：

| 时间 | 动作 | 责任主角 |
| --- | --- | --- |
| 周一 | 更新样本集、补新失败案例、冻结本周评测范围 | 测试/质量 |
| 周二到周三 | 跑评测、收集分数、整理代表样本 | 测试/质量 + 工程 |
| 周四 | 复盘会，确定 Top 问题与归因 | 测试/质量 + 产品 + 工程 + 业务 |
| 周五 | 输出周报，确认改进行动和下轮验证点 | 测试/质量 |

## 10. 质量保障分层

V1 不只关注发布前离线评测，也需要为发布时和发布后的质量闭环预留位置。

| 层次 | 机制 | 目的 |
| --- | --- | --- |
| 发布前 | 评估数据集、自动化评测、回归评测、阈值检查 | 确保已知场景不过线不发布 |
| 发布时 | 输入 Guard、输出 Guard、格式校验、兜底策略 | 拦截明显风险和不可接受输出 |
| 发布后 | 采样评估、用户反馈、日志分析、趋势监控 | 发现未知问题并补充 Eval |
| 持续改进 | 线上问题入库、修复后复测、回归集沉淀 | 让每次失败都变成长期资产 |

V1 第一阶段先重点建设“发布前 + 持续改进”两层；发布时 Guard 和发布后采样评估纳入后续自动化待办。

## 11. 三阶段演进

### 阶段 1：手工化

- 样本维护以人工为主
- 评测执行可以是脚本加手工整理
- 报告和复盘由人工汇总

适用条件：

- 刚起步
- 样本量不大
- 团队需要先建立共同语言

### 阶段 2：半自动化

- 样本结构化存储
- grader 可批量执行
- 周报和基础看板自动生成
- 固定调度周期

适用条件：

- 已跑通至少一个试点
- 样本与 rubric 相对稳定
- 团队希望降低运营成本

### 阶段 3：智能化

- 自动聚类失败模式
- 自动发现分数漂移
- 自动推荐可能根因
- 自动扩充回归集

适用条件：

- 多个试点复用同一体系
- 有较稳定的样本和结果历史
- 组织愿意建设统一能力

## 12. V1 治理原则

- 不把 V1 做成“打分即决策”的黑盒
- 每一轮评测都必须有代表性样本供人工复核
- judge 不替代业务真相，只用于提效
- 失败模式必须沉淀为结构化标签，避免散落在会议纪要里
- 结果报告必须能映射到具体行动项，不能只停留在分数层面
- 每次线上问题修复后，必须判断是否沉淀为 Eval 样本或回归样本

## 13. V1 验收标准

### 框架层

- 团队成员能仅凭文档说清 `评什么、谁负责、多久跑一次、结果怎么用`
- 三类操作手册边界清晰，不互相混用

### 运行层

- 质量团队能独立组织一次完整评测闭环
- 结果能形成明确修复项和责任人

### 复用层

- 新试点可以在不重新定义字段的前提下复用模板
- 第二个业务团队接入时，不需要改动总框架主体
