# RAG/问答操作手册

## 1. 适用场景

适用于以下类型的 AI 能力：

- 内部知识问答
- 检索增强问答
- 政策、制度、SOP、产品说明类问答
- 需要回答时同时给出依据或引用的问答系统

以下场景建议改用其他手册：

- 明显偏内容创作的长文生成
- 需要多轮工具调用和状态改变的复杂 agent
- 无检索依赖的纯模型回答

## 2. 评估对象

RAG/问答建议拆成检索层和生成层分别评估，不要把“检索 + 生成”混成一个总分。

### 检索层

关注：

- 是否命中正确文档
- 是否召回关键证据
- 是否混入明显噪声
- 是否存在权限越界或引用错误

### 生成层

关注：

- 是否忠于引用上下文
- 是否回答了用户真正的问题
- 是否覆盖关键点
- 是否表达清晰、结构可用
- 是否出现幻觉、空答或误导

## 3. 核心指标

| 维度 | 定义 | 推荐评分方式 |
| --- | --- | --- |
| 命中率 | 检索结果中是否包含正确文档 | code/human |
| 召回感知 | 检索前 K 条是否覆盖关键证据 | human/llm |
| 知识召回质量 | 检索 chunks 是否包含回答问题所需的关键信息 | llm/human |
| 忠实度 | 答案是否严格基于上下文 | llm/human |
| 相关性 | 答案是否正面回应用户问题 | llm |
| 完整性 | 关键点是否答全 | llm/human |
| 清晰度 | 表达是否结构化、易理解 | llm |
| 安全/边界 | 是否越权、误导、错误暴露敏感信息 | code/human |

## 4. 推荐 grader 组合

### 4.1 Code grader

优先承担确定性、低争议检查：

- 输出格式是否正确
- 是否包含引用
- 引用是否存在
- 是否为空答
- 是否存在越权文档引用
- 是否触发明显敏感词或规则

### 4.2 LLM-as-judge

重点覆盖难以规则化的质量维度，建议分维度独立判断：

- 相关性
- 忠实度
- 完整性
- 清晰度
- 知识召回质量

执行建议：

- 每个维度单独一条 rubric
- judge 提供 `分数 + 理由 + 是否需要人工复核`
- judge 可以输出 `Unknown`，防止强行判断
- 知识召回质量只看检索结果是否足够，不混入最终回答质量

### 4.3 人工抽检

用于：

- 校准 LLM judge
- 审核边界样本
- 审核高风险问题
- 验证评测结果与真实用户体验是否一致

## 5. 样本集设计

### 初始规模

- 第一版建议 `50-100 条`
- 少于 30 条时，结果波动过大，不建议作为稳定基线

### 样本来源优先级

1. 真实用户问题
2. 历史失败案例
3. 补充边界题

### 结构配比建议

| 类型 | 占比建议 | 说明 |
| --- | --- | --- |
| 常见题 | 40% | 高频真实问题 |
| 难题 | 20% | 需要跨文档理解或高上下文依赖 |
| 边界题 | 20% | 模糊、信息不足、歧义场景 |
| 应拒答/应澄清题 | 20% | 不能胡答的场景 |

### 样本来源要求

不要只用工程师设计的问题。样本来源建议至少包含：

- 真实用户问题
- 线上翻车案例
- 线上监控发现的异常表达
- 主动构造的故障注入样本，例如空检索、旧文档、权限不足、相似文档干扰

## 6. 失败模式 taxonomy

建议固定以下一级标签，便于跨轮对比：

- `knowledge_missing`：知识源缺失或过期
- `retrieval_miss`：关键文档未召回
- `retrieval_noise`：噪声文档干扰
- `prompt_misaligned`：Prompt 导致表达或判断偏差
- `model_hallucination`：生成脱离上下文
- `answer_incomplete`：回答关键点缺失
- `answer_unclear`：回答难理解或组织不当
- `boundary_failure`：拒答、澄清、权限边界处理错误

## 7. RAG 问题快速诊断

| 低分指标 | 可能原因 | 首选改进方向 |
| --- | --- | --- |
| 知识召回质量低 | 相关知识存在但没召回 | Query 改写、提高 top_k、混合检索、调整切片 |
| 命中率低 | 正确文档排序靠后或未进入结果 | 引入 reranker、优化 metadata、过滤知识域 |
| 忠实度低 | 答案脱离上下文 | 强化引用约束、要求逐点基于证据回答 |
| 完整性低 | 只答了部分关键点 | 补充 rubric、增加关键点检查 |
| 边界控制低 | 空数据或无权限时仍然编造 | 增加拒答/澄清样本和输出 Guard |

## 8. 启动步骤

1. 确定一个边界清晰的知识域，不覆盖全库。
2. 用成功标准模板定义业务目标、场景和评分维度。
3. 建立首批黄金集并完成基础标注。
4. 编写 code grader 和 judge rubric。
5. 先做 dry-run，再进入固定周节奏。
6. 每周都产出 Top failure modes 和行动项。

## 9. 每周输出要求

至少包含：

- 总分与分维度分数
- 代表性失败样本
- Top failure modes
- 根因归类
- 行动项、责任人、下轮验证点

## 10. 进入回归阶段的条件

满足以下条件后，可把部分能力评测提升为回归集：

- 连续两轮样本结构稳定
- judge 与人工的一致性达到团队认可阈值
- 已经完成至少一轮“发现问题 → 调整 → 再评测验证”的闭环
- 高频失败模式已沉淀为稳定 grader 或样本标签
