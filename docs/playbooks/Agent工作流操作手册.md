# Agent 工作流操作手册

## 1. 适用场景

适用于以下类型的 AI 能力：

- 多步任务执行
- 工具调用和外部系统交互
- 需要改变环境状态的流程自动化
- 有明确终态或任务完成标志的 agent

## 2. 评估对象

Agent 场景的评估对象优先是“结果和终态”，其次才是轨迹过程：

- 任务是否完成
- 环境终态是否正确
- 工具调用是否有效
- 异常时是否恢复
- 轨迹是否出现明显无效循环

## 3. 核心指标

| 维度 | 说明 | 推荐 grader |
| --- | --- | --- |
| 任务成功率 | 是否完成任务目标 | state/human |
| 状态正确率 | 系统最终状态是否符合预期 | state |
| 工具调用有效性 | 工具是否用对、参数是否合理 | tool/state |
| 工具调用匹配度 | 工具名、参数、顺序是否符合任务需要 | tool/transcript |
| 异常恢复能力 | 遇错后是否能纠偏或回退 | transcript/human |
| 轨迹稳定性 | 是否出现无效循环、过长路径、明显偏航 | transcript/llm |

## 4. 推荐 grader 组合

- 环境状态检查：验证最终状态或副作用
- 工具调用检查：验证关键工具是否被合理使用
- 轨迹抽检：人工或 llm 审查高风险轨迹
- 人工校准：用于复杂任务和边界场景

## 5. 工具调用评估

Agent 的最终回答正确，不代表工具调用过程正确。V1 建议对有工具能力的 Agent 增加 `tool_call_match` 类评估：

| 检查项 | 说明 | 示例 |
| --- | --- | --- |
| 工具名 | 是否选择了正确工具 | 查询天气应调用天气工具，而不是知识库搜索 |
| 工具参数 | 参数是否完整、格式是否正确 | 城市、日期、ID、金额等是否提取正确 |
| 调用顺序 | 多步任务是否按合理顺序执行 | 先查库存再下单，先查权限再写入 |
| 必要调用 | 必须调用的工具是否被调用 | 涉及实时数据时不能只凭模型记忆回答 |
| 多余调用 | 是否调用了不必要或高风险工具 | 不需要写入时不应调用更新接口 |

工具调用评估不应强制所有任务走固定路径，但关键安全动作、状态变更动作和事实查询动作需要有明确检查项。

## 6. 样本类型建议

Agent 样本集建议按 4 类组织：

| 类型 | 说明 |
| --- | --- |
| 正常 | 标准流程和常见任务 |
| 边缘 | 信息不完整、表达模糊、权限边界、复杂组合任务 |
| 混沌 | 工具超时、空数据、接口异常、脏数据、提示注入 |
| 记忆或上下文 | 多轮上下文、历史状态、前后依赖 |

第一版不追求数量大，优先沉淀真实失败样本和高风险链路。

## 7. 试点启动步骤

1. 选择一个边界清晰、终态明确的任务集。
2. 为每类任务定义成功终态和失败终态。
3. 建立多轮样本，记录任务输入、环境起点、预期终态、样本来源和案例类型。
4. 先验证 state grader，再补轨迹 grader。
5. 运行复盘时优先归因“在哪一步偏了”，同步检查最终失败结果。

## 8. 复盘重点

- 是任务定义不清，还是 agent 执行偏了
- 是环境限制导致失败，还是模型策略问题
- 是关键工具没用到，还是用了但参数错了
- 是否存在可稳定复现的失败步骤
